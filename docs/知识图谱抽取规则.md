# Neo4j 知识图谱抽取规则

本文档描述企业知识平台当前知识图谱抽取规则。规则已经落地到 `backend/app/services/neo4j_graph_service.py`，并由 `backend/tests/test_neo4j_knowledge_graph.py` 覆盖。

## 一、实体类型体系

| 类型 | 定义 | 识别特征 | 示例 |
| --- | --- | --- | --- |
| `file` | 文档或文件实体 | 带常见文件扩展名，来自文件名或正文提及 | `企业知识平台最终方案总结.docx` |
| `api` | 接口实体 | `GET/POST/PUT/PATCH/DELETE + /api/...` 或 `/api/...` 路径 | `POST /api/v1/workspaces/{workspace_id}/documents` |
| `technology` | 技术组件实体 | 数据库、中间件、框架、模型、协议和服务名 | `Neo4j`、`Milvus`、`MinIO` |
| `knowledge_base` | 知识库实体 | 以知识库、向量库结尾，或别名归一到知识库对象 | `个人知识库`、`企业知识库` |
| `workspace` | 工作区实体 | 个人工作区、企业工作区、工作空间等 | `企业工作区` |
| `module` | 模块实体 | 系统、平台、模块、服务、中心、驾驶舱、工作台 | `知识库模块` |
| `process` | 流程实体 | 上传、解析、切片、入库、同步、检索等动作流程 | `文档解析`、`向量入库` |
| `config` | 配置实体 | 大写环境变量或配置键 | `RERANK_MODEL_PATH` |
| `permission` | 权限安全实体 | 权限、角色、安全、隔离、认证、审计、策略 | `权限策略` |
| `concept` | 概念实体 | 其他有语义价值的业务概念 | `研究领域` |

## 二、实体抽取规则

1. 标题、章节标题、前四行段首、表格单元格、代码/API 行的权重高于普通正文。
2. 文件名、API 路径、环境变量、技术组件优先作为强实体保留。
3. 表格单元格会进入候选池，用于构建“接口 -> 服务”等结构化关系。
4. 项目符号列表不会被误判为标题，标题下的列表项会作为标题节点的子实体。
5. 常见别名会合并为同一节点，例如：
   - `Neo4j 图数据库`、`图数据库` -> `Neo4j`
   - `Milvus 向量库`、`向量数据库` -> `Milvus`
   - `MinIO 对象存储`、`对象存储服务` -> `MinIO`
   - `个人空间` -> `个人工作区`
6. `workspace_id`、`document_id`、`user_id` 等字段名不作为核心图谱节点，避免抢占关系主语。

## 三、关系抽取规则

1. 显式谓词优先：`包含`、`包括`、`涵盖`、`依赖`、`属于`、`引用`、`调用`、`写入`、`隔离`、`保护`、`检索` 等会生成关系边。
2. 同一句中有多个谓词时，按谓词片段切分，只连接当前片段内的对象，避免把后半句全部挂到第一个谓词上。
3. 安全类关系如 `隔离`、`保护` 会优先选择工作区、知识库、模块、文件或 API 作为主语，而不是 `workspace_id` 这类字段。
4. 标题下项目符号列表会生成结构包含关系，例如 `知识库模块 -> 文档解析`。
5. 表格会按表头推断关系：
   - 表头包含 `依赖` 或 `服务`：生成 `依赖`
   - 表头包含 `调用`、`API` 或 `接口`：生成 `调用`
   - 表头包含 `引用`：生成 `引用`
   - 表头包含 `属于` 或 `归属`：生成 `属于`
   - 表头包含 `包含`、`组成`、`模块`、`子项`：生成 `包含`
6. 不再使用“上下文相邻”作为兜底关系，避免生成孤立、弱证据边。

## 四、权重与置信度规则

1. 节点权重范围为 `1-100`，关系权重范围为 `1-100`。
2. 节点权重考虑：
   - 出现位置：标题、章节、段首、表格、代码行
   - 提及频率
   - 实体类型优先级
   - 是否多次出现并可形成关系
3. 关系权重考虑：
   - 谓词强度：`隔离`、`调用`、`依赖`、`属于` 等强关系权重更高
   - 源节点和目标节点自身权重
   - 源/目标类型，例如 API、文件、知识库、工作区、技术组件会提升关系权重
4. 低质量关系不会低于可展示阈值，但没有明确证据的关系不会生成。

## 五、检索与展示规则

1. 每个工作区只查询当前 `workspace_id` 下的节点和关系。
2. 每个知识库对应文档产生的实体关系会带有 `document_id`，前端可按知识库组合查看。
3. 前端优先使用 `entity_type` 展示中文图例和颜色；旧数据没有 `entity_type` 时回退到 `semantic_type`。
4. 2D 和 3D 图谱展示同一批真实实体关系，不使用对话、文档、片段等占位节点伪造图谱主体。
